01

What this system does

SRIA RT is a pipeline for detecting lateral-movement behavior in enterprise authentication telemetry using graph-structural features rather than signature matching. It does not look for known malware names, process lists, or static indicators. It looks for patterns in how authentication events connect — source hosts, destination hosts, users, timing, and novelty — that are structurally consistent with lateral movement even when individual events appear benign.

The pipeline produces a ranked analyst review queue: a prioritized list of suspicious authentication episodes for human examination. It is not an automated blocking system. Every output is intended for analyst review, not autonomous action.

Core hypothesis Real lateral movement leaves graph-structural traces — first-time edges, fanout patterns, compact bursts — even when static indicators are unavailable. SRIA RT tests that hypothesis.

04

Architecture

The pipeline operates in three layers. The feature engine extracts structural signals from authentication topology. The learned ranker orders accepted episodes by likelihood of containing lateral movement. The packaging layer organizes ranked output into analyst-readable review queues grouped by source cluster.

LayerComponentRole
1 — Feature EngineSRIA authentication topologyExtracts graph-structural signals from auth events
2 — Candidate GatesHand-written gate logicGenerates accepted episode candidates; not the final ranker
3 — Learned Rankerrf_depth10_cw_noneRanks accepted episodes by lateral-movement likelihood
4 — Output LayerDeployment queue + cluster packagingAnalyst-readable review queues grouped by source

The hand-written gate score is not the final ranking authority. After leakage-corrected baseline comparison, the learned ranker (RF depth-10, trained on v0.3.3, scored on v0.3.6) consistently outperformed the hand score. The hand gates remain useful as candidate generators and explanation metadata, not as ranking logic.

Official recall is computed from the deduplicated union of represented red-team event indices across the queue. Cluster debug annotation totals (which may double-count events attached to multiple episodes) are documented separately and labeled debug_rt_event_annotation_total. The delta between annotation totals and official recall is 2 events — a 1.5% annotation overlap — confirmed by v0.5.5 diagnostic.

Key finding The SRIA feature engine is valuable. The hand-written SRIA score is weak as a final ranker. The learned RF depth-10 ranker, trained with leakage-stripped features, recovered 100% of represented red-team events within the top 500 accepted-episode review queue. The hand score found zero represented red-team events in the top 1,000.
06

Known limitations

Review-queue ranking only, not deployment precision Metrics cover accepted episodes, not full enterprise background
Sparse-window evaluation constraint Evaluation used red-team time windows; broader negative-only evaluation needed
LANL-specific validation External dataset validation not yet performed
Negative-background false positive rate unknown How many high-scoring episodes appear per day in normal traffic is unknown
Model stability across splits RF depth-10 not yet tested across time-based or cross-branch splits
Interpretability tradeoff RF depth-10 outperforms but is less directly interpretable than depth-6 tree
Not full SRIA v3.0 Semantic policy, delegation, and approval provenance layers not implemented
Adversary generalization Real attackers may use side channels, novel tooling, or insider access not modeled here

What remains to be done

The most important next step is negative-background evaluation — running the pipeline against authentication traffic outside the sparse red-team windows to measure analyst burden in normal enterprise conditions. This is the question any serious reviewer will ask first and the one currently unanswered.
